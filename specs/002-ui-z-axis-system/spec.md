# Feature Specification: UI组件Z轴坐标体系

**Feature Branch**: `002-ui-z-axis-system`  
**Created**: 2026-03-22  
**Status**: Draft  
**Input**: User description: "当前渲染采用组件树。组件树中每个节点有x坐标，y坐标。现在我需要你给他们添加一个z坐标。每个节点添加一个枚举配置，用来标明开启z轴体系/legacy体系。如果开启z轴体系，则对其子组件开启z轴体系：展示&点击事件响应等全部使用z轴体系进行高低覆盖。否则legacy体系，还是沿用旧的行为。"

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Z轴层次渲染 (Priority: P1)

作为游戏UI开发者，我希望组件能够使用Z轴坐标控制渲染层次，以便实现复杂的UI叠加效果（如弹出菜单覆盖在主界面上方、模态对话框遮挡下方内容等）。

**Why this priority**: 这是Z轴体系的核心价值，解决当前组件树只能依赖渲染顺序控制层次的问题，提供更灵活的UI布局能力。

**Independent Test**: 可以独立测试：创建两个重叠的组件，通过设置不同的Z坐标值，验证Z值高的组件始终渲染在Z值低的组件上方。

**Acceptance Scenarios**:

1. **Given** 两个组件A和B在相同X、Y位置且大小重叠，**When** 组件A的Z坐标为10，组件B的Z坐标为5，**Then** 组件A渲染在组件B上方（遮挡组件B）
2. **Given** 一个父组件启用了Z轴体系，**When** 渲染其子组件时，**Then** 系统按照Z坐标值从低到高排序渲染
3. **Given** 多个子组件具有相同的Z坐标值，**When** 渲染时，**Then** 保持它们在组件树中的原始顺序（稳定性保证）

---

### User Story 2 - Z轴点击事件响应 (Priority: P1)

作为游戏UI开发者，我希望点击事件能够根据Z轴坐标正确分发到最上层的组件，以便实现准确的UI交互（如点击弹窗时不穿透到下方的按钮）。

**Why this priority**: 与渲染层次同样重要，确保用户交互行为符合视觉预期。如果点击事件不能正确处理Z轴层次，将导致用户误操作。

**Independent Test**: 可以独立测试：创建两个重叠的可点击组件，点击重叠区域，验证只有Z值最高的组件接收到点击事件。

**Acceptance Scenarios**:

1. **Given** 两个可点击组件A和B在相同位置重叠，Z(A)=10, Z(B)=5，**When** 用户点击重叠区域，**Then** 只有组件A接收到点击事件
2. **Given** 一个模态对话框（高Z值）覆盖在可点击的按钮（低Z值）上方，**When** 用户点击对话框覆盖按钮的区域，**Then** 按钮不响应点击事件
3. **Given** 组件A启用了Z轴体系且包含子组件B和C，**When** 处理点击事件时，**Then** 系统先检查子组件的Z坐标，再决定是否向父组件传递

---

### User Story 3 - 向后兼容的Legacy模式 (Priority: P1)

作为现有游戏项目的维护者，我希望在添加Z轴功能的同时保持向后兼容，以便现有代码无需修改即可继续工作。

**Why this priority**: 这是功能推广的关键。如果不保持向后兼容，所有现有UI代码都需要修改，迁移成本过高。

**Independent Test**: 可以独立测试：创建使用旧代码风格的组件树，验证在不启用Z轴体系的情况下，渲染和点击行为与之前完全一致。

**Acceptance Scenarios**:

1. **Given** 一个组件使用Legacy模式（不启用Z轴体系），**When** 渲染其子组件时，**Then** 按照组件树遍历顺序渲染，忽略Z坐标
2. **Given** 一个Legacy模式的组件包含可点击子组件，**When** 处理点击事件时，**Then** 按照原有的命中测试顺序处理
3. **Given** 一个混合场景（部分组件启用Z轴，部分使用Legacy），**When** Legacy组件与Z轴组件重叠时，**Then** Legacy组件的行为保持不变

---

### User Story 4 - Z轴体系的继承传播 (Priority: P2)

作为UI框架使用者，我希望父组件启用Z轴体系后，其子组件自动继承该体系，以便简化配置并确保一致性。

**Why this priority**: 提升开发效率，避免在每个子组件上重复配置。P2因为即使不实现，手动配置也能工作。

**Independent Test**: 可以独立测试：创建一个启用Z轴体系的父组件，添加子组件但不显式配置，验证子组件自动使用Z轴体系。

**Acceptance Scenarios**:

1. **Given** 父组件启用Z轴体系，**When** 添加子组件时，**Then** 子组件自动继承Z轴体系配置
2. **Given** 父组件启用Z轴体系且包含多层嵌套子组件，**When** 渲染整个树时，**Then** 所有后代组件都使用Z轴体系
3. **Given** 子组件显式设置了与父组件不同的体系配置，**When** 渲染时，**Then** 子组件使用自己的配置（允许覆盖）

---

### Edge Cases

- **相同Z坐标处理**: 当多个组件Z坐标相同时，如何确定渲染顺序？建议保持组件树遍历的稳定性
- **Z坐标负值**: 是否允许负Z坐标？建议允许，用于表示位于基准平面下方的元素
- **混合体系边界**: 当一个Z轴体系组件的子组件是Legacy模式，且两者重叠时，如何处理渲染顺序？
- **点击事件穿透**: 当Z轴最高的组件不消费点击事件时，是否向下方组件传递？建议可配置
- **性能考虑**: 大量组件时，按Z坐标排序的性能影响？建议考虑使用空间分割或缓存排序结果
- **动态Z坐标变化**: 运行时修改Z坐标是否需要立即重新排序？建议在下一次渲染周期应用

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 组件节点必须支持Z坐标属性（浮点类型），默认值为0
- **FR-002**: 每个组件节点必须包含一个枚举配置项，标明使用Z_AXIS_MODE或LEGACY_MODE
- **FR-003**: 当组件配置为Z_AXIS_MODE时，其子组件默认继承Z_AXIS_MODE（除非显式覆盖）
- **FR-004**: 当组件配置为LEGACY_MODE时，渲染和事件处理保持原有行为（组件树遍历顺序）
- **FR-005**: Z_AXIS_MODE下，渲染时必须按照Z坐标从低到高排序后绘制
- **FR-006**: Z_AXIS_MODE下，点击事件必须分发到Z坐标最高且命中测试通过的组件
- **FR-007**: Z坐标相同的情况下，保持组件树遍历的稳定性（先遍历的优先或后遍历的优先需明确定义）
- **FR-008**: 必须提供API允许运行时动态修改组件的Z坐标
- **FR-009**: 必须提供API允许运行时切换组件的坐标体系模式（Z_AXIS_MODE/LEGACY_MODE）

### Quality Requirements *(与宪法 I. 代码质量 对齐)*

- **QR-001**: 代码必须遵循 P3C 规范，零 ERROR 级别警告
- **QR-002**: 公共 API 必须包含完整的 JavaDoc 注释，说明Z坐标行为和模式切换影响
- **QR-003**: 方法长度不得超过 50 行，圈复杂度不得超过 10
- **QR-004**: 坐标体系相关代码必须清晰分离，避免与现有代码深度耦合

### Testing Requirements *(与宪法 II. 测试标准 对齐)*

- **TR-001**: 单元测试覆盖率必须达到 80%（核心业务 ≥90%）
- **TR-002**: 必须包含渲染顺序测试，验证不同Z坐标下的正确绘制顺序
- **TR-003**: 必须包含点击事件测试，验证Z轴层次正确的事件分发
- **TR-004**: 必须包含向后兼容测试，验证LEGACY_MODE与原有行为一致
- **TR-005**: 必须包含混合模式测试，验证Z轴组件与Legacy组件共存时的行为

### User Experience Requirements *(与宪法 III. 用户体验一致性 对齐)*

- **UX-001**: 交互必须符合既有 UI 模式（左键选择、右键菜单等）
- **UX-002**: 用户操作必须在 100ms 内提供视觉/音频反馈
- **UX-003**: Z轴体系切换必须是显式配置，避免意外改变现有UI行为
- **UX-004**: API设计必须直观，Z坐标语义与行业惯例一致（数值大表示在前方）

### Performance Requirements *(与宪法 IV. 性能要求 对齐)*

- **PR-001**: Z坐标排序不得导致帧率低于 60 FPS
- **PR-002**: 1000个组件的Z坐标排序时间不得超过 1ms
- **PR-003**: 点击事件处理增加的开销不得超过原有时间的 20%
- **PR-004**: 内存占用增量：每个组件增加Z坐标和模式枚举的开销必须最小化

### Key Entities *(include if feature involves data)*

- **Component**: UI组件基类，新增Z坐标属性和坐标体系模式配置
- **CoordinateSystemMode**: 枚举类型，定义Z_AXIS_MODE和LEGACY_MODE两种模式
- **ComponentTree**: 组件树管理器，负责根据模式选择合适的渲染和事件处理策略
- **ZIndexSorter**: Z坐标排序器，负责按Z坐标对组件进行排序

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 开发者可以在不修改现有代码的情况下继续使用Legacy模式，现有测试100%通过
- **SC-002**: 启用Z轴体系的组件，其渲染顺序100%按照Z坐标从低到高执行
- **SC-003**: 启用Z轴体系的组件，点击事件100%分发到Z坐标最高且命中的组件
- **SC-004**: 包含1000个组件的场景，Z坐标排序处理时间小于1ms
- **SC-005**: 新功能的单元测试覆盖率达到90%以上
