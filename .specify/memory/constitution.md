# xenoamess's-civilization 项目宪法

## 核心原则

### I. 代码质量（不可协商）

所有代码必须遵循统一的编码标准，确保可读性、可维护性和一致性。

- **代码风格统一**：所有 Java 代码必须遵循项目定义的代码格式（P3C 规范），使用统一的缩进、命名规范和注释标准
- **复杂度控制**：单个方法不得超过 50 行代码，圈复杂度不得超过 10；类长度不得超过 500 行
- **静态代码分析**：所有提交必须通过 P3C 代码检查，零警告容忍（ERROR 级别必须修复，WARNING 级别需评审）
- **代码审查**：所有代码变更必须经过至少一名其他开发者审查通过后方可合并
- **文档要求**：所有公共 API 必须包含 JavaDoc 注释，说明功能、参数、返回值和异常

**设计理由**：游戏项目生命周期长，代码质量直接影响长期维护成本。统一的代码风格降低团队沟通成本，静态分析提前发现潜在缺陷。

### II. 测试标准（不可协商）

全面的测试覆盖是保证游戏稳定性和可维护性的基础。

- **单元测试覆盖率**：核心业务逻辑代码单元测试覆盖率必须达到 80% 以上，关键路径（游戏状态管理、存档系统、网络同步）必须达到 90%
- **测试先行**：新增功能必须先编写测试用例，确保测试失败后再实现功能（Red-Green-Refactor）
- **测试分类**：
  - 单元测试：测试独立模块，毫秒级执行，无外部依赖
  - 集成测试：测试模块间交互，验证数据流和状态一致性
  - 契约测试：验证 API 和事件协议的前后兼容性
- **自动化执行**：所有测试必须在 CI/CD 流水线中自动执行，测试失败阻止合并
- **测试命名规范**：测试方法名必须清晰描述测试场景，使用 `given_when_then` 或 `should_xxx` 格式

**设计理由**：回合制策略游戏逻辑复杂，测试是防止回归缺陷的唯一有效手段。高覆盖率确保重构安全，自动化保障持续交付质量。

### III. 用户体验一致性

所有功能和界面必须遵循统一的用户体验设计原则，确保玩家获得连贯、直观的游戏体验。

- **交互一致性**：相同类型的操作必须使用相同的交互模式（如左键选择、右键菜单、拖拽移动）
- **视觉一致性**：UI 组件遵循统一的设计语言（颜色、字体、间距、动画时长），使用项目 UI 组件库
- **反馈即时性**：所有玩家操作必须在 100ms 内提供视觉或音频反馈；耗时操作必须显示进度指示
- **错误处理**：错误信息必须用户友好，说明问题原因和解决方案，避免技术术语和堆栈跟踪暴露给玩家
- **无障碍支持**：UI 必须支持键盘导航，关键功能提供快捷键；字体大小可调节，色盲友好配色
- **本地化**：所有用户可见文本必须外部化，支持多语言切换

**设计理由**：策略游戏学习曲线陡峭，一致的体验降低认知负担。即时反馈增强游戏手感，良好的错误处理避免玩家困惑。

### IV. 性能要求

游戏必须在目标硬件上保持流畅运行，提供稳定的帧率和响应速度。

- **帧率标准**：游戏主循环必须稳定维持 60 FPS（16.67ms/帧），最低不低于 30 FPS
- **内存预算**：
  - 基础内存占用：≤ 512MB（启动后静态资源）
  - 运行时峰值：≤ 2GB（大型地图、后期游戏）
  - 无内存泄漏：连续运行 4 小时内存增长 ≤ 10%
- **加载时间**：
  - 游戏启动：≤ 10 秒
  - 存档加载：≤ 5 秒
  - 场景切换：≤ 2 秒
- **渲染性能**：单帧渲染 draw call ≤ 500，贴图内存 ≤ 512MB
- **性能监控**：集成性能分析工具，定期生成性能报告；关键版本必须通过性能基准测试

**设计理由**：策略游戏后期数据量大，性能问题会随游戏进度恶化。明确的性能预算指导技术选型和优化决策，确保所有玩家获得流畅体验。

## 附加约束

### 技术栈规范

- **语言**：Java 17+（LTS 版本）
- **构建工具**：Maven 3.8+
- **游戏引擎**：cyan_potion 引擎（项目内部维护）
- **依赖管理**：优先使用成熟稳定的开源库，引入新依赖需技术评审

### 安全要求

- 存档文件必须校验完整性，防止篡改
- 多人游戏通信必须加密，防止中间人攻击
- 用户输入必须验证，防止注入攻击

## 开发工作流

### 代码审查要求

- 所有 PR 必须通过 CI 检查（编译、测试、代码规范）
- 所有 PR 必须至少一名审查者批准
- 重大变更（架构调整、公共 API 修改）需技术方案评审

### 质量门禁

- 代码合并前：单元测试通过率 100%，覆盖率不低于基准
- 版本发布前：完整集成测试通过，性能基准测试通过
- 每次提交：静态代码分析零 ERROR

### 版本管理

- 遵循语义化版本规范（SemVer）：MAJOR.MINOR.PATCH
- 破坏性变更必须标记，提供迁移指南
- 保留 CHANGELOG，记录所有版本变更

## 治理

本宪法是项目开发的最高指导原则，优先于其他实践和约定。

### 修订流程

- 宪法修订需提交书面提案，说明修改理由和影响范围
- 重大原则变更（新增、删除、重新定义）需项目维护者一致同意
- 小幅澄清和措辞优化需至少两名维护者批准
- 所有修订必须更新版本号并记录在修订历史中

### 合规审查

- 每个功能实现的规划阶段必须检查宪法合规性（Constitution Check）
- 代码审查时需验证是否违反任何原则，如有违反必须说明理由
- 定期（每季度）审查宪法有效性，根据项目演进调整
